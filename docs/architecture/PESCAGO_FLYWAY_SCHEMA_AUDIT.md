## 2. Inventario de entidades y esquema inferido

> **Leyenda de defaults**
>
> - **Default aplicación (Java):** valor asignado en constructor vacío o lógica de dominio; **no implica** `DEFAULT` en PostgreSQL.

- **FK físicas:** ninguna declarada en JPA (`@ManyToOne` / `@JoinColumn` ausentes).

## 4. Hallazgos de discrepancias o incertidumbres

| # | Hallazgo | Impacto Flyway |
|---|----------|----------------|
| 1 | **`ddl-auto=update` activo** en main | Hibernate puede crear/alterar tablas sin versionado; divergencia silenciosa entre entornos |
| 2 | **Sin Flyway/Liquibase/SQL** | No hay fuente de verdad del esquema fuera del runtime Hibernate |
| 3 | **Tabla `entreprenuers`** (typo) | `V1__initial_schema.sql` debe respetar nombre legacy exacto |
| 4 | **Columnas embeddable** (`length`/`width`/`height`, `vehicle_brand`/`plate`/`driver`) | **Confirmado en BD real:** aplanadas sin prefijo; no usar `dimensions_*` ni `carrier_data_*` en `V1` |
| 5 | **Pluralización custom** (`Pluralize` library) | Nombre de tablas depende de reglas de librería, no de `@Table` |
| 6 | **`status` VARCHAR mezclado histórico** | BD puede contener español (`Pendiente`) e inglés (`PENDING`, canónicos nuevos); Flyway no debe asumir valores uniformes |
| 7 | **Sin FKs físicas** | `V1` puede omitir FKs (alineado al plan §6.4) o añadirlas como decisión explícita posterior |
| 8 | **Sin índices en entidades** | Hibernate `update` no crea índices de consulta (`findByCarrierId`, etc.); rendimiento y `V1` deben definirlos explícitamente si se desean |
| 9 | **Tipos monetarios `Double`** | No coinciden con regla v2 `NUMERIC(12,2)` futura; fuera de alcance 0D pero relevante para migraciones posteriores |
| 10 | **`receipts` almacena PAN/CVV** | Esquema actual incluye columnas sensibles; migración futura separada (plan §6.3) |
| 11 | **Auditoría `java.util.Date` vs `TIMESTAMPTZ`** | Plan v2 propone `TIMESTAMPTZ`; esquema Hibernate actual probablemente `timestamp without time zone` |
| 12 | **Perfil `local` no redefine datasource** | Desarrollo local sigue usando PostgreSQL + `ddl-auto=update` salvo override manual |

## 6. Riesgos específicos de migrar desde `ddl-auto=update`

1. **Deriva entre entornos:** dev/staging/prod pueden tener columnas distintas según orden de arranque y versiones de entidades.
2. **Flyway + `update` simultáneos:** doble fuente de verdad; riesgo de conflictos al habilitar Flyway sin desactivar `update`.
3. **Baseline incorrecto en BD existente:** marcar `baseline` antes de alinear `V1` puede bloquear correcciones futuras.
4. **`flyway clean` accidental:** mitigar con `spring.flyway.clean-disabled=true` (plan §6.1).
5. **Pérdida de datos en embeddables:** renombrar columnas `length`/`width`/`height` o `vehicle_brand`/`plate`/`driver` sin migración explícita rompe lecturas (nombres reales confirmados en verificación read-only).
6. **Estados legacy en `status`:** migración de datos a canónicos es **opcional y posterior**; no mezclar con `V1` estructural.
7. **Arranque de tests H2 no valida PostgreSQL:** `create-drop` en H2 no garantiza paridad con producción.

## 7. Estrategia propuesta por escenario

### 7.1 Base PostgreSQL nueva (greenfield — Escenario A §6.1)

1. Crear BD vacía `pescago` (o nombre de env).
2. Aplicar `V1__initial_schema.sql` vía Flyway (cuando exista y esté validado).
3. Mantener `ddl-auto=update` **temporalmente** solo en dev hasta verificar paridad, luego pasar a `validate` o `none`.
4. Evolución: `V2__...` en adelante (columnas v2 del plan §6.3).

### 7.2 Base PostgreSQL existente (Escenario B §6.1)

1. **Backup obligatorio** antes de cualquier cambio.
2. Exportar esquema real (`pg_dump --schema-only`) y comparar con inventario §2.
3. Ajustar borrador `V1__initial_schema.sql` para que sea **documento de referencia**, no script ciego de aplicación.
4. Si esquema coincide: `flyway baseline` en versión 1 **después de revisión humana**.
5. **No** usar `baseline-on-migrate=true` indiscriminadamente.
6. Primera migración evolutiva real: **`V2__...`** (nunca recrear tablas existentes).
7. `ddl-auto=validate` solo cuando Flyway y entidades coincidan.

### 7.3 Desarrollo local

1. Documentar dos modos en README/procedimiento:
   - **Modo A:** PostgreSQL local + Flyway (objetivo 0D).
   - **Modo B:** Solo tests Maven con H2 (sin sustituir validación PostgreSQL).
2. Perfil `local` futuro: considerar datasource dedicado o flags Flyway explícitos (sin cambiar aún en 0D).
3. **No arrancar** `spring-boot:run` contra BD compartida hasta definir estrategia baseline.

### 7.4 Pruebas H2 actuales

- Mantener H2 para tests de integración existentes (auth, status, carrier v2).
- H2 usa `create-drop` + naming strategy igual → útil para regresión JPA, **insuficiente** para validar Flyway PostgreSQL.
- Paridad embeddables/H2 `MODE=PostgreSQL`: aceptable para tests de dominio; no para DDL Flyway.

## 11. Decisión recomendada: H2 vs PostgreSQL vs Testcontainers

| Enfoque | Recomendación | Motivo |
|---------|---------------|--------|
